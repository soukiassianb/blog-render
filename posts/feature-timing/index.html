Feature Timing

Feature Timing

One of the well known pitfalls when building a product is the bias toward adding features as a way to improve the overall product experience and build engagement. Much has been written about why this approach rarely works and an opposing, more purist school of thought take pride in building products with minimalists feature sets.

When determining if one should build a feature, the discussion often fall around the tension between some sort of static set of beliefs: the minimalist designer who wants to keep things simple *vs *the “built it and they will come” engineer who wants to add new features to please a vocal but marginal set of users (and possibly increase complexity).

This tension often overshadows another variable in the equation that is based in a more objective reality: The timing of the feature within a product’s lifecycle.

Primary features determine your product

In the early days of a product, the features should provide and support the core interaction of the product: If you’re building a photo-sharing app, the features you build should allow the act of sharing a picture, the act of consuming this picture or enhance the “value unit”, the picture shared.

Instagram is a good example, the primary features of Instagram are: the picture sharing tool, the feed, and the filters. Those features were implemented day one for good reasons:

  • The “sharing picture” actions support the core interaction of a “producer” of content (sharing a picture, easy).

  • The feed support the core interaction of a “consumer” of content (discovering & consuming pictures).

  • The filters enhance the core value units, the pictures shared (which creates value for both parties, consumer & producers).

Primary features determine the nature of a product. The question then is not “should we build this feature for our product” but rather “what is our product ?”.

The tricky features are the ones which comes later on, which we’ll call “secondary”. They are tricky because they are the ones for which timing matters most.

Secondary features (come in different types)

  1. They create value on top of a core value unit

Medium is a place to write on the internet. The core value units of Medium are the stories. One example of a secondary feature which build value on top of the stories are Collections, which have recently been introduced.

The ability to browse Medium by theme is valuable for users, but can only exist if there are enough good stories that can be classified. That’s why it did not make sense to build Collections day one.

Features that build value on top of core value units are dependent on the number and quality of those. That’s why those features should generally not be built at launch but later on when the quantity and quality of the value units start to grow.

2. They support the core Interaction

Another type of secondary features are the ones that support or encourage the core interaction.

Let’s take our Instagram example again, liking and commenting pictures are secondary features which add a social layer on top of the core interaction and create a feedback loop that reinforce the users’ motivation to post more pictures, in order to get likes and comments.

Those features can drive engagement but are tricky because they can become so intrinsically linked to the core interaction that they may fool you into thinking that your core interaction is valuable enough. Once there it’s hard to decipher what really drives the usage, as a consequence your may add more and more of those kind of features and make your product more complex.

How many people use Instagram for the love of photography ? If Instagram were to drop the likes and comments features, what percentage of users would stop using the app ? Hard to say.

Don’t get me wrong, I’m not saying likes and comments are bad features to launch day one. But social incentives or gamification should not be the primary drivers of usage of a product (well, except if you’re building a social network or a game).

If you are trying to determine whether the core interaction of your product creates enough value for customers, you’d probably better off waiting a bit before adding too much of those.

Peripheral features

Those are the non-essential. They often are features that support indirectly the core interaction, or directly the secondary features.

They are non-essential from a product perspective, but sometimes be strategically significant for the parent company.

To keep our Instagram example: Instagram Direct, the app’s version of a private chat was probably launched to avoid users having to go over another app for private messaging. It does support and expand Instagram’s social layer, but it is not an essential part of the product.

Peripheral features can be side-bets, strategic moves or non-essential but often appreciated luxuries likes customization options.

When talking about new features, one should try to determine where it falls on the primary — secondary — peripheral spectrum and prioritize accordingly.

Features are not fixed on the spectrum, secondary features may become primary or primary features secondary, users are the only judge. There is a solid argument to make about Instagram’s filter not being as much core to the user experience than the social layer is today.

Having a clear vision for a product is great, but feature timing is what really matters. Building too much too fast is rarely a good strategy. Having a more focused and iterative approach helps you learn and make steadier steps.
